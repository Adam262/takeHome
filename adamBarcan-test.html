  • Q: Describe the differences between the IE6, IE7, and W3C box models.

    CSS Box Model

    The CSS box model represents each html element on a page as a box with four possible properties: content, border, padding and margin. The below browser models differ on how these properties contribute to an element's height or width.

    IE6 box model

    In the IE6 box model, an element's default height or width property reflects the total combined height or width of its content, left and right borders and left and right padding. You can think of IE6 width as an element's visible height or width. For example, a div with width: 300px will continue to have a 300px width, even if a border or padding are added. Only the addition of margin would add to the element's width.

    IE7 box model

    The IE7 box model adopts the approach of the W3C box model, as described below.

    W3C box model

    In the W3C box model, an element's default width property reflects the width of its content only. Border, padding or margin are then added to content width. One can rever to IE6 box-model by styling an element with "box-sizing: border-box."

    Describe the MVC design pattern as it applies to web servers - start from initial HTTP request and end with writing the response.

    MVC Design Pattern

    Overview

    Model-View-Controller (MVC) is a design pattern that separates an app's files by their role. An MVC pattern's directories comprise: (i) models, which access a data store, (ii)views, whose HTML, JS and CSS files form the user interface and (iii) controllers, which contain the app's logic. Good MVC organizes code in a way that is logical, modular, easy to maintain and easy for new developers to grasp.

    HTTP request and response over an MVC-designed app

    HTTP requests typically start from user actions in the view. For example, a user may sign up, log in submit a form or click a link that requests a page. Let's use login as an example -- the user's login name and hashed password are sent to a back-end GET route contained in a controller file. This file contains functions that handle typical requests relating to a user -- ie, POST, or create, GET, or read, PUT, or update, and DELETE. In this case, our GET route's function reads user data from an object contained in the request. The same controller logic may contain a query to a database - eg, to look up the user by an ID, also container in the request body. Although the database query may be part of controller logic, the actual schema for the user's profile properties sit in a model directory -- ie, a directory of files that access a data base. For example, there is a schema containing properties associated with a user -- id, name, hashed password, etc. The model now communicates user status back to the controller - eg, by a JSON object. If a user is verified, the controller may serve a use profile back to the view. If a user is not verified, the controller may serve an error message such as 401.

    Describe the pros and cons of version control.

    Version Control

      PROS
    • Version control enables groups to develop code together regardless of the group's size or geographic distribution. Proper version control even allows managers of large open-source projects to accept features from diverse users while protecting the core code's integrity.
    • Version control allows even solitary users to track changes to a project over time. Rather than compare changes across members of a group, you can compare versions of your own program over time. You can add features as new branches -- only merge to the main branch when you have tested new features for bugs. And you can always revert to a working version if a thorny bug appears after merging new branches.
      CONS
    • Potential for introducing bugs if good admin practices not followed -- ie, good review of pull requests
    • Potential for revealing sensitive code, such as business logic or API keys -- again, if good practices not followed
